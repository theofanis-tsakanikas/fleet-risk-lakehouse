# ADR-008: Real-World Data via VED Trip Replay

**Date:** 2026-07  
**Status:** Accepted

---

## Context

Every claim this platform makes — the ±60s temporal join, the risk score, drift
monitoring, DQ quarantining, masking — was demonstrated on data generated by the
project's own mock producers. A pipeline that only ever processes data its own codebase
fabricated proves engineering mechanics, but nothing about behaviour on data with real
statistical structure: genuine GPS traces, real speed profiles, actual hard-braking
events. That circularity was the project's honest weak point.

Fully real *paired* data does not exist publicly: no open dataset couples a driver's
wearable stream with their vehicle's telemetry.

## Decision

Add a **replay layer** (`src/replay/`) that streams **real vehicle telemetry** through
the exact same Bronze → Silver → Gold contract, from the **Vehicle Energy Dataset**
(VED: 383 vehicles, ~374k miles, Ann Arbor 2017–18, Apache-2.0 — see
`data/ved/README.md`):

- A committed **real sample** (`data/ved/ved_sample.csv`: 10 vehicles, 18 trips selected
  for valid GPS and genuine hard-braking events) makes the default replay reproducible
  with zero downloads; `scripts/fetch_ved.py` pulls the full dataset for larger runs.
- **Trackers are real**: GPS positions, speeds and timestamps come from the VED trace,
  resampled to the fleet cadence (60 s), pseudonymised onto the `DRV_xx` roster (no VED
  identifier reaches the lakehouse), and rebased onto a replay anchor — each driver's
  trips play back-to-back from the anchor, so the fleet is immediately active.
- **Biometrics are simulated — conditioned on the real driving events**, and documented
  as such (`src/replay/biometrics.py`): hard-braking events detected in the raw trace
  (> 8 km/h/s decel) trigger an exponentially-decaying heart-rate/stress response, and
  overspeed (threshold from `RISK_MODEL` — the same source of truth as the score and
  alerts) sustains elevation. Deterministic per (driver, trip, seed).
- **One bundle, two jobs**: `fleet_replay_job` mirrors the monitoring job's medallion
  DAG (YAML aliases of the same task definitions), swapping only the generator tasks.
  Both replay tasks receive `{{job.start_time.iso_datetime}}` as the anchor, so the two
  streams align tick-for-tick inside the join window by construction. A 30-minute
  periodic trigger (PAUSED by default, `var.replay_pause_status`) turns the lakehouse
  into a continuously refreshing system when enabled.

The mock producers remain: they exist to exercise the Silver cleansing rules with
synthetic *dirt* (sentinels, ghost drivers, duplicates), which clean real data cannot do.
Real data proves the happy path is real; mock data proves the defences work.

## Consequences

**Benefits**

- The join, risk scoring, and drift monitoring now demonstrably work on trips that
  actually happened; an end-to-end test replays the real sample through the production
  Silver transforms into a non-empty Gold enriched view.
- Honesty is preserved: what is real (vehicle telemetry) and what is modelled
  (biometrics, fuel proxy) is stated in code, docs, and the data README.
- Determinism is kept everywhere (seeded biometrics, sorted assignment, anchored
  timestamps), so the replay is as testable as the mocks.

**Trade-offs**

- The biometric stream is still synthetic — the honest ceiling of public data. The
  conditioning model is deliberately simple (documented gains/decay), not physiological
  research.
- The committed sample adds ~1.2 MB to the repo; the full dataset stays out (gitignored,
  fetched on demand).
- VED locations are de-identified by the dataset authors (geo-fencing/fog), so
  map-accurate street-level analytics are approximate by design.
